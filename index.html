Why we're leaving Nuxt

we ran Nuxt with SSR off — a server framework as a router

What we actually used

  • File-based routing
  • Plugins & layouts
  • Auto-imports
  • $fetch wrapper

What we paid for it

  • SSR machinery we never render
  • Magic that agents must guess
  • Major-version upgrade treadmill
  • Debugging through abstraction layers

Everything in the left column exists as a small, explicit Vue library.

ما Nuxt را همیشه در حالت SPA اجرا می‌کردیم — یعنی یک فریم‌ورک سروری که فقط به عنوان روتر استفاده می‌شد. چیزهایی که واقعاً استفاده می‌کردیم چهار قابلیت ساده است و همه‌شان معادل صریح و سبک در اکوسیستم Vue دارند. در عوض هزینه‌ی جادوی framework، آپگریدهای بزرگ و دیباگ لایه‌به‌لایه را می‌دادیم. برای agentها هم «جادو» یعنی حدس‌زدن — و حدس یعنی باگ.

II

Choosing the UI framework

four candidates · six criteria · one open decision

پرده‌ی دوم: مهم‌ترین تصمیم امروز. چهار کاندیدا را با شش معیار می‌سنجیم، به یک پیشنهاد می‌رسیم — و تصمیم نهایی را همین جلسه با هم می‌گیریم.

Our use case = our criteria

Pure-Vue first-class

Must work without Nuxt — no half-supported mode.

RTL · Farsi · Jalali

RTL layouts, Persian digits, Jalali dates — not an afterthought.

Design-token theming

CSS-variable tokens that map 1:1 to Figma.

Component depth

Data table, date picker, uploads, overlays — the boring 80%.

TS + a11y quality

Typed props/slots, ARIA correctness out of the box.

Agent-friendliness

LLM-readable docs, predictable APIs, official skills.

The last one is new — and it's a first-class criterion, because agents write most of our UI code.

قبل از مقایسه باید معیار داشته باشیم. پنج‌تای اول کلاسیک‌اند: کار بدون Nuxt، پشتیبانی واقعی RTL و تقویم جلالی، توکن‌های طراحی مبتنی بر CSS variable که با Figma یک‌به‌یک map شوند، عمق کامپوننت‌ها، و کیفیت TypeScript و دسترس‌پذیری. معیار ششم جدید است: agent-friendliness. وقتی بیشترِ کد UI را agent می‌نویسد، مستندات قابل‌خواندن برای LLM و API قابل‌پیش‌بینی دیگر «امتیاز مثبت» نیست — معیار درجه‌یک است.

Candidate 1 · Nuxt UI v4

Reka UI primitives + Tailwind CSS v4 — despite the name, a Vue library

Strengths

  • Standalone Vue via official Vite plugin
  • Theming is Tailwind @theme tokens
  • v4 merged Pro — 100+ components, free
  • 50+ locales, RTL supported
  • llms.txt docs + official agent skill

Watch out

  • Commits us to Tailwind utilities
  • Younger than Vuetify / PrimeVue
  • Name confuses newcomers — docs say Vue-ready

already running in front-cp-v2 production

کاندیدای اول: Nuxt UI نسخه‌ی ۴. اسمش گمراه‌کننده است — روی Reka UI و Tailwind ساخته شده و رسماً بدون Nuxt، با یک Vite plugin در Vue خالص کار می‌کند. مدل theming آن دقیقاً همان design token است: Tailwind @theme و CSS variableها. نسخه‌ی ۴ کامپوننت‌های Pro را هم رایگان کرد. مستندات llms.txt دارد و skill رسمی‌اش را هم در skills-lock پروژه pin کرده‌ایم. ریسک‌ها: وابستگی به Tailwind و جوان‌تر بودن نسبت به رقبا. و مهم‌تر از همه: الان در production ما واقعاً کار می‌کند.

Candidate 2 · Vuetify 4

the incumbent — Material Design, batteries included

Strengths

  • Huge, mature component set
  • First-class RTL for years
  • Blueprints + defaults system
  • Our team knows it deeply

Watch out

  • Material look — hard to de-Google
  • SASS-variable theming, not CSS-native tokens
  • v3 → v4 (MD3) migration cost lands on us
  • Heavier runtime; we fight defaults today
کاندیدای دوم: Vuetify — همان چیزی که الان در اپ Nuxt v4 داریم. نقاط قوت واقعی‌اند: مجموعه‌ی عظیم کامپوننت، RTL قدیمی و پایدار، و دانش عمیق تیم. اما مشکل‌ها را هم از نزدیک می‌شناسیم: ظاهر Material که برای برند ما باید مدام override شود، theming مبتنی بر SASS که با رویکرد توکن‌محور CSS ما نمی‌خواند، و مهاجرت v3 به v4 که خودش یک پروژه است. ما داریم از Nuxt خارج می‌شویم که با default ها نجنگیم — نه این‌که جنگ را به لایه‌ی UI منتقل کنیم.

Candidate 3 · PrimeVue 4

the component powerhouse — 90+ widgets, token-themed

Strengths

  • Best-in-class DataTable
  • v4 theming = real design-token API
  • Styled and unstyled/Tailwind modes
  • Broadest widget coverage

Watch out

  • Mixed internals — parts still Options API / JS
  • Two theming systems to learn
  • RTL fine, Farsi/Jalali ecosystem thinner
کاندیدای سوم: PrimeVue. اگر فقط یک معیار داشتیم — عمق کامپوننت — برنده بود؛ DataTable آن بهترینِ اکوسیستم است. نسخه‌ی ۴ یک API واقعی design token هم دارد و هم حالت styled و هم unstyled با Tailwind. اما داخل کتابخانه هنوز یکدست نیست — بخشی از کامپوننت‌ها Options API و JavaScript هستند و در حال مهاجرت‌اند — و دو سیستم theming یعنی دو چیز برای یادگرفتن agentها.

Candidate 4 · shadcn-vue

not a dependency — the CLI copies component source into your repo

Strengths

  • You own every line — agents can read & patch
  • Reka UI primitives + Tailwind (same base as Nuxt UI)
  • CSS-variable theming by construction
  • Zero lock-in, zero upgrade treadmill

Watch out

  • You own every line — maintenance too
  • Data table = assemble-it-yourself (TanStack)
  • Fewer ready composites; community port
کاندیدای چهارم: shadcn-vue — پارادایم متفاوت. کتابخانه نصب نمی‌کنید؛ CLI سورس کامپوننت را داخل repo کپی می‌کند. برای agentها این رویاست: کل کد جلوی چشم است، قابل خواندن و قابل patch. ولی همان مزیت، هزینه است: نگهداری همه‌ی آن کد با ماست، جدول داده باید با TanStack سرهم شود، و پروژه پورتِ جامعه است نه محصول رسمی. برای تیم ما که ۸۰٪ نیازش کامپوننت‌های آماده است، این trade-off نمی‌صرفد — اما از فلسفه‌اش استفاده می‌کنیم.

III

Components, classified

our four layers · the Atomic Design standard · migrate or not?

پرده‌ی سوم: طبقه‌بندی کامپوننت‌ها. اول سیستم خودمان، بعد مقایسه با استاندارد صنعت — Atomic Design — و یک تصمیم باز: مهاجرت بکنیم یا نه. هدف نهایی: هیچ agent و هیچ توسعه‌دهنده‌ای هرگز نپرسد «این کامپوننت را کجا بسازم؟»

The four layers

Base* Global primitives — shared/ui/base/. Wrap Nuxt UI + heavy libs. Props/emits only. BaseInput, BaseDataTable, BaseFilePond.
App* Shell chrome — shared/ui/shell/. Header, sidebar, nav, user menu. Knows layout, not domains. AppSidebar, AppUserMenu.
{Module} components Dumb domain UI — modules/X/presentation/components/. Domain-shaped props, no stores, no API. MediaCard, PaymentSidebar.
{Module} screens Smart containers — modules/X/presentation/screens/. Orchestrate use-cases + stores, pass data down. CampaignListScreen.

Only screens are smart. Everything else renders props.

چهار لایه داریم. Base* یعنی primitiveهای سراسری که Nuxt UI و کتابخانه‌های سنگین را wrap می‌کنند — فقط props و emits. App* یعنی پوسته‌ی اپ: هدر، سایدبار، منو — layout را می‌شناسد اما دامنه را نه. کامپوننت‌های ماژول، UI گنگِ دامنه‌اند: props دامنه‌شکل می‌گیرند اما نه store دارند نه API. و فقط لایه‌ی چهارم — screenها — هوشمند است: use-case و store را به هم وصل می‌کند و داده را به پایین پاس می‌دهد.

Heavy libs live behind wrappers

BaseFilePond

FilePond + tus resumable uploads — one component, one config.

BaseRichTextEditor

TipTap + useRichTextEditor — toolbar decided once.

BaseServerDataTable

Table + server pagination/sort state, URL-synced.

  • Consumers never import the third-party API
  • Swap or upgrade the lib in one file
  • Agents learn one project API, not five vendor docs
کتابخانه‌های سنگین هرگز مستقیم استفاده نمی‌شوند. FilePond، TipTap، جدول‌ها — همه پشت یک wrapper پروژه‌ای. سه دلیل: مصرف‌کننده هیچ‌وقت API فروشنده را import نمی‌کند؛ تعویض یا آپگرید کتابخانه فقط یک فایل را عوض می‌کند؛ و agent به‌جای پنج داکیومنتِ متفاوت، فقط یک API پروژه‌ای را یاد می‌گیرد. این همان جایی است که فلسفه‌ی shadcn را نگه داشتیم: سطح API مال ماست.

The industry standard — Atomic Design

Brad Frost, 2013 — the shared vocabulary of big-tech design systems

Atoms Indivisible primitives — button, input, label, icon.
Molecules A few atoms, one job — search field = label + input + button.
Organisms Self-contained sections — header, product grid, sidebar.
Templates Page skeletons — layout and slots, no real content.
Pages Templates with real data — what users actually see.

classifies by composition: each level is built from the one below · echoed across Material, Carbon, Lightning, Polaris

تا این‌جا سیستم خودمان را دیدید؛ سؤال منصفانه: دنیا چه می‌کند؟ استاندارد رایج صنعت Atomic Design است — برد فراست، ۲۰۱۳. ایده از شیمی می‌آید: atom ها primitive های تجزیه‌ناپذیرند؛ molecule ترکیب چند atom با یک وظیفه؛ organism بخش خودکفای صفحه؛ template اسکلت بدون محتوا؛ و page همان template با داده‌ی واقعی. این واژگانِ مشترک design system های بزرگ است — از Material گوگل تا Carbon و Lightning و Polaris.

Atomic vs ours — a different axis

Atomic level Our equivalent Match?
Atoms Base* primitives — BaseInput, BaseChip ≈ direct
Molecules Base* composites — BaseDatePicker, BaseServerDataTable ≈ we don't split these two levels
Organisms App* shell + module components ≈ but we also split by ownership
Templates layouts/ — default, auth, blank ≈ direct
Pages screens (smart) + pages/ routes ⚠ atomic has no smart/dumb rule

Atomic's axis

Composition granularity — "how big is it?" Designer-first vocabulary.

Our axis

Ownership & data access — "who may import it, may it fetch?" Agent-first guardrails.

نگاشت تقریباً یک‌به‌یک وجود دارد: atom و molecule هر دو در Base* ما هستند — ما این دو سطح را جدا نکرده‌ایم؛ organism معادل App* و کامپوننت‌های ماژول است اما ما مالکیت را هم جدا می‌کنیم؛ template همان layout ماست و page همان screen به‌علاوه‌ی route. تفاوت اصلی محور است: اتمیک بر اساس اندازه‌ی ترکیب دسته‌بندی می‌کند و طراح‌محور است؛ سیستم ما بر اساس مالکیت و دسترسی به داده — و برای agent ها ساخته شده. اتمیک درباره‌ی smart/dumb، ایزوله بودن ماژول‌ها و جای fetch حرفی نمی‌زند.

Adopt it? — a decision for this meeting

Migration path — if yes

  • Split shared/ui/base → atoms/ + molecules/
  • Shell + module components → organisms/
  • layouts/ = templates · screens = pages — already 1:1
  • Update rules, skills, lint zones — agents relearn once

Does it answer our use case?

  • Gives: designer-shared vocabulary, Figma alignment, familiar onboarding
  • Misses: data-access rules, module isolation, smart/dumb — our real guardrails
  • Net: a naming layer, not an architecture — our rules stay either way

Open — adopt Atomic · hybrid · keep ours

hybrid = keep the four layers, speak Atomic vocabulary in Labs & Figma

اگر مهاجرت کنیم مسیر روشن است: base به atoms و molecules تقسیم می‌شود؛ پوسته و کامپوننت‌های ماژول organism می‌شوند؛ layout ها همین حالا template اند و screen ها page — نگاشت تقریباً یک‌به‌یک است و هزینه‌ی اصلی به‌روزکردن rule ها و skill ها و zone های lint است. اما آیا به use case ما جواب می‌دهد؟ می‌دهد: واژگان مشترک با طراح‌ها و Figma و onboarding آشنا. نمی‌دهد: قواعد دسترسی به داده، ایزوله بودن ماژول و smart/dumb — یعنی guardrail های واقعی ما. جمع‌بندی برای بحث: اتمیک بیشتر لایه‌ی نام‌گذاری است تا معماری. سه گزینه: مهاجرت کامل، حالت ترکیبی (لایه‌های خودمان + واژگان اتمیک در Labs و Figma)، یا ماندن. همین‌جا تصمیم بگیریم.

IV

Where logic lives

composables · stores · utils — each has exactly one job

پرده‌ی چهارم: منطق کجا زندگی می‌کند؟ سه خانه داریم — composable، store و util — و هرکدام دقیقاً یک وظیفه دارند.

Three homes, one rule each

Composable

Reusable reactive behavior. Returns refs, computed, functions. State lives with the caller — a new copy per use.

shared/composables/use*.ts

Store (Pinia)

Shared state that outlives routes. One singleton, devtools, persistence. The exception, not the default.

app/stores/*.ts

Util

Pure function. No reactivity, no Vue import, same input → same output. Trivially testable.

shared/lib/*.ts

+ Module use-case

A composable that orchestrates one domain workflow — fetch, rules, state for its screen. modules/X/application/use-cases/useCampaignList.ts

سه خانه، هرکدام یک قانون. Composable یعنی رفتار reactive قابل استفاده‌ی مجدد — هر بار صدا زدن، یک نسخه‌ی تازه؛ state مال صدازننده است. Store یعنی state مشترکی که از route عبور می‌کند — singleton است و باید استثنا باشد، نه پیش‌فرض. Util یعنی تابع خالص: نه reactivity، نه import از Vue؛ ورودی یکسان، خروجی یکسان. و شکل چهارم که در واقع همان composable است اما در لایه‌ی application ماژول: use-case — ارکستراسیون یک جریان دامنه برای screen خودش.

Instant review rejections

Fetch in a dumb component

Components render props. Fetching belongs to use-cases.

Store as universal cache

Route-local state goes in the use-case or the URL — not Pinia.

Util that imports Vue

The moment it needs a ref, it's a composable. Move it.

A second HTTP client

All traffic goes through useCustomFetch. No raw fetch/axios.

چهار الگویی که در review مستقیم رد می‌شوند — و lint هم بیشترشان را می‌گیرد: fetch داخل کامپوننت گنگ؛ استفاده از Pinia به‌عنوان کش همه‌چیز درحالی‌که state محلی باید در use-case یا URL باشد؛ util ای که Vue import می‌کند — همان لحظه باید composable شود؛ و ساختن HTTP client دوم — همه‌ی ترافیک از useCustomFetch رد می‌شود، بدون استثنا.
