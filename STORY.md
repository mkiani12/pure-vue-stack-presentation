# متن ارائه — استک جدید: Vue خالص

این فایل، همراهِ ارائه‌ی `index.html` است. روی اسلایدها عمداً متن کم است؛
داستان کامل و متن گفتاری ارائه این‌جاست. هر بخش سه قسمت دارد:
**روی صفحه** (چیزی که مخاطب می‌بیند)، **متن ارائه** (چیزی که می‌گویید) و
**پل** (جمله‌ی اتصال به اسلاید بعد).

خط داستانی کلی: *چرا می‌رویم ← پایه‌ی جدید ← بزرگ‌ترین تصمیم (فریم‌ورک UI)
← نظم کامپوننت‌ها ← نظم منطق ← سیستم طراحی زنده ← و این‌که هیچ‌کدام به
حافظه‌ی آدم‌ها تکیه ندارد.*

---

## اسلاید ۱ — Pure Vue: استک جدید

**روی صفحه:** عنوان، تگ‌های استک، اشاره به این‌که تصمیم فریم‌ورک UI امروز گرفته می‌شود.

**متن ارائه:**
سلام به همه. امروز درباره‌ی مهم‌ترین تغییر فنی فرانت‌اندمان صحبت می‌کنیم:
مهاجرت از Nuxt به **Vue خالص**. این جلسه یک معرفی نیست — یک جلسه‌ی تصمیم
است. چهار موضوع را می‌بندیم: انتخاب فریم‌ورک UI از بین چهار کاندیدا،
طبقه‌بندی کامپوننت‌ها، این‌که هر تکه‌ی منطق کجا زندگی کند
(composable یا store یا util)، و در نهایت Labs و design tokenها.

یک فرض هم در کل ارائه جاری است: توسعه‌ی ما **agent-driven** است. یعنی
بیشترِ کد را Claude و Codex می‌نویسند و ما معماری و قواعد را تعیین می‌کنیم.
هر تصمیمی که امروز می‌گیریم باید برای یک agent هم به همان اندازه واضح باشد
که برای یک انسان.

**پل:** اما اول باید به یک سؤال جواب بدهیم — چرا اصلاً داریم از Nuxt خارج می‌شویم؟

---

## اسلاید ۲ — چرا از Nuxt خارج می‌شویم

**روی صفحه:** دو کارت — «چیزی که استفاده می‌کردیم» و «هزینه‌ای که می‌دادیم».

**متن ارائه:**
واقعیت ساده است: ما Nuxt را همیشه با SSR خاموش اجرا کرده‌ایم. یعنی یک
فریم‌ورک سمت سرور را فقط به‌عنوان روتر و اسکافولد استفاده می‌کردیم.

چیزهایی که واقعاً از Nuxt مصرف می‌کردیم چهار چیز بود: روتینگ فایل‌محور،
سیستم plugin و layout، auto-import و wrapper مربوط به `$fetch`. همین.

در عوض چه هزینه‌ای می‌دادیم؟ ماشین SSR کاملی که هرگز render نمی‌کند.
جادوی فریم‌ورک — آن چیزهایی که «خودش کار می‌کند» تا روزی که کار نمی‌کند و
باید سه لایه abstraction را دیباگ کنی. و چرخه‌ی آپگریدهای major که هر بار
برنامه‌ریزی و ریسک می‌خواهد.

برای agentها این هزینه دوبرابر است: جادو یعنی رفتاری که در کد پیدا نمی‌شود
و agent باید حدس بزند. حدس یعنی باگ.

نکته‌ی کلیدی این اسلاید: **تمام ستون چپ، معادل صریح و کوچک در اکوسیستم
خود Vue دارد.** پس چیزی از دست نمی‌دهیم.

**پل:** بریم ببینیم این جایگزین‌ها دقیقاً چه هستند.

## اسلاید ۳ — پرده‌ی اول: پایه

**روی صفحه:** جداکننده‌ی پرده‌ی I.

**متن ارائه:**
پرده‌ی اول: پایه‌ی جدید. قولی که این‌جا می‌دهیم این است: هر قابلیت Nuxt با
چیزی جایگزین می‌شود که بشود **کدش را خواند**.

**پل:** نقشه‌ی یک‌به‌یک جایگزینی، اسلاید بعد.

## اسلاید ۴ — نقشه‌ی جایگزینی Nuxt ← Vue خالص

**روی صفحه:** جدول قابلیت‌به‌قابلیت.

**متن ارائه:**
این جدول قلب پرده‌ی اول است. سطر به سطر:

- **روتینگ فایل‌محور** را `unplugin-vue-router` می‌دهد — همان تجربه‌ی
  Nuxt، به‌علاوه‌ی مسیرهای کاملاً typed. یعنی اگر route ای وجود نداشته
  باشد، TypeScript همان لحظه خطا می‌دهد.
- **پلاگین‌ها** توابع ساده در `src/app/plugins/` می‌شوند، با ترتیب نصب
  صریح. دیگر «کشف خودکار» نداریم؛ ترتیب همان است که در کد نوشته شده.
- **middleware** همان navigation guard خود vue-router است — یک فایل guard،
  بدون قرارداد جادویی.
- **layoutها** یک کامپوننت ۳۰ خطی به اسم `LayoutHost` است که خودمان
  نوشته‌ایم و مال خودمان است.
- **auto-import را عمداً کنار می‌گذاریم.** import صریح یعنی هم reviewer و
  هم agent با یک نگاه می‌فهمند هر فایل به چه چیزهایی وابسته است.
- **`$fetch`** — نکته‌ی جالب: زیر `$fetch` همیشه کتابخانه‌ی `ofetch` بوده.
  ما همان را مستقیم برمی‌داریم و policy خودمان (auth، retry، خطا) را در
  `useCustomFetch` دورش می‌کشیم.
- **`runtimeConfig`** هم با `import.meta.env` و یک فایل typed جایگزین
  می‌شود که در build چک می‌شود.

**پل:** حالا کل استک را یک‌جا ببینیم.

## اسلاید ۵ — دیوار استک

**روی صفحه:** هشت کارت: Vue 3.5، Vite 6، Pinia 3، vue-router 4، vue-i18n 10، VeeValidate + Zod، TypeScript max-strict، bun.

**متن ارائه:**
این استک کامل ماست. چند نکته:

- TypeScript در سخت‌گیرترین حالت است: `strict`، `noUncheckedIndexedAccess`،
  `exactOptionalPropertyTypes` و ممنوعیت مطلق `any`. این سخت‌گیری برای
  agentها بهترین دوست است — کامپایلر نیمی از review را انجام می‌دهد.
- فرم‌ها با VeeValidate و Zod ساخته می‌شوند؛ یعنی اعتبارسنجی و type از یک
  schema می‌آیند.
- i18n از روز اول دوزبانه (فارسی/انگلیسی) و RTL-first است.

و مهم‌ترین جمله‌ی این اسلاید: **این لیست تئوری نیست.** این دقیقاً استکی است
که در `front-cp-v2` — بازنویسی داشبورد درنا — همین الان در production کار
می‌کند. ما داریم درباره‌ی چیزی تصمیم می‌گیریم که یک‌بار آزمایشش کرده‌ایم.

**پل:** پایه محکم است. حالا بزرگ‌ترین تصمیم: روی این پایه، با کدام
فریم‌ورک UI می‌سازیم؟

## اسلاید ۶ — پرده‌ی دوم: انتخاب فریم‌ورک UI

**روی صفحه:** جداکننده‌ی پرده‌ی II — «چهار کاندیدا، شش معیار، یک حکم».

**متن ارائه:**
پرده‌ی دوم، مهم‌ترین تصمیم امروز. چهار کاندیدا را بررسی می‌کنیم:
Nuxt UI، Vuetify، PrimeVue و shadcn-vue. اول معیارها را می‌چینیم، بعد هر
کاندیدا را منصفانه می‌سنجیم، بعد جدول امتیاز، و در نهایت حکم.

**پل:** هر مقایسه‌ای بدون معیار، سلیقه است. اول معیارها.

## اسلاید ۷ — معیارهای ما

**روی صفحه:** شش کارت معیار.

**متن ارائه:**
معیارها مستقیم از use case خودمان درمی‌آیند:

۱. **پشتیبانی درجه‌یک از Vue خالص.** فریم‌ورکی که فقط «در تئوری» بدون
Nuxt کار کند قبول نیست.

۲. **RTL و فارسی و تقویم جلالی.** محصول ما فارسی‌زبان است؛ RTL باید
شهروند درجه‌یک باشد، نه پچ.

۳. **Theming مبتنی بر design token.** توکن‌های CSS variable که یک‌به‌یک با
Figma ما map شوند — این پیش‌شرط پرده‌ی پنجم است.

۴. **عمق کامپوننت.** جدول داده، date picker جلالی‌پذیر، آپلود، overlay —
همان ۸۰ درصد خسته‌کننده‌ای که نباید خودمان بسازیم.

۵. **کیفیت TypeScript و دسترس‌پذیری.** props و slot های typed و ARIA درست.

۶. و معیاری که در مقایسه‌های کلاسیک پیدا نمی‌کنید: **agent-friendliness.**
مستنداتی که برای LLM قابل مصرف باشد (مثل llms.txt)، API قابل پیش‌بینی، و
وجود skill رسمی. وقتی بیشتر کد UI را agent می‌نویسد، این دیگر امتیاز جانبی
نیست — معیار درجه‌یک است و ما آگاهانه به آن وزن می‌دهیم.

**پل:** با این شش خط‌کش، برویم سراغ کاندیدای اول.

---

## اسلاید ۸ — کاندیدای ۱: Nuxt UI v4

**روی صفحه:** کارت قوت‌ها / کارت هشدارها؛ پانوشت «در production ما اجرا می‌شود».

**متن ارائه:**
کاندیدای اول Nuxt UI نسخه‌ی ۴ است و باید همین اول یک سوءتفاهم را رفع کنیم:
**اسمش گمراه‌کننده است.** Nuxt UI روی primitiveهای Reka UI و Tailwind CSS
ساخته شده و به‌طور رسمی، با یک Vite plugin مخصوص
(`@nuxt/ui/vue-plugin`)، در پروژه‌ی Vue خالص کار می‌کند — با auto-import،
theming کامل و TypeScript کامل.

قوت‌ها: مدل theming آن *عیناً* همان چیزی است که ما می‌خواهیم — Tailwind
`@theme` و CSS variable؛ یعنی فایل `tokens.css` ما بدون ترجمه، زبان بومی
این فریم‌ورک است. نسخه‌ی ۴ کامپوننت‌های Pro را با هسته یکی و رایگان کرد —
بالای صد کامپوننت. به بیش از ۵۰ زبان ترجمه شده و RTL را پشتیبانی می‌کند.
و برای agentها: مستندات `llms.txt` دارد و ما skill رسمی‌اش را در
`skills-lock.json` پروژه pin کرده‌ایم.

هشدارها: ما را به Tailwind متعهد می‌کند (که البته با رویکرد توکنی ما
همسوست)، و از Vuetify و PrimeVue جوان‌تر است.

و مهم‌ترین شاهد: `front-cp-v2` همین حالا با آن در production است — با
RTL فارسی واقعی.

**پل:** کاندیدای دوم را خوب می‌شناسیم — چون الان داریم با آن زندگی می‌کنیم.

## اسلاید ۹ — کاندیدای ۲: Vuetify 4

**روی صفحه:** کارت قوت‌ها / هشدارها.

**متن ارائه:**
Vuetify گزینه‌ی فعلی ماست و باید با آن منصف باشیم. مجموعه‌ی کامپوننتش
عظیم و پخته است، RTL را سال‌هاست درجه‌یک پشتیبانی می‌کند، سیستم Blueprint
و defaults دارد و — مزیت واقعی — تیم ما آن را عمیق بلد است. Vuetify نسخه‌ی
۴ هم (بهمن‌ماه گذشته / فوریه ۲۰۲۶) با Material Design 3 منتشر شده.

اما دردها را هم از نزدیک می‌شناسیم:

- **ظاهر Material.** برند ما گوگلی نیست؛ در اپ فعلی مدام در حال override
  کردن ظاهر پیش‌فرضیم.
- **Theming با متغیرهای SASS** در زمان build است، نه توکن‌های CSS-native.
  با خط لوله‌ی توکنی Figma-محور ما نمی‌خواند.
- **مهاجرت v3 به v4** به‌خاطر MD3 پر از breaking change است — این هزینه
  در هر صورت روی میز ماست، حتی اگر بمانیم.
- و runtime سنگین‌تری دارد.

جمله‌ی کلیدی: ما داریم از Nuxt خارج می‌شویم که با پیش‌فرض‌های فریم‌ورک
نجنگیم. منطقی نیست همان جنگ را در لایه‌ی UI ادامه بدهیم.

**پل:** کاندیدای سوم، غول کامپوننت‌ها.

## اسلاید ۱۰ — کاندیدای ۳: PrimeVue 4

**روی صفحه:** کارت قوت‌ها / هشدارها.

**متن ارائه:**
اگر مسابقه فقط یک معیار داشت — تعداد و عمق کامپوننت — PrimeVue برنده بود.
بالای ۹۰ کامپوننت دارد و `DataTable` آن بهترینِ کل اکوسیستم Vue است.
نسخه‌ی ۴ هم از نظر theming جدی شده: یک API واقعی design token دارد و دو
حالت ارائه می‌دهد — styled با توکن‌ها، و unstyled که با Tailwind خودتان
استایل می‌دهید.

اما سه هشدار: داخل کتابخانه هنوز یکدست نیست — بخشی از کامپوننت‌ها هنوز
Options API و JavaScript هستند و طی امسال در حال مهاجرت به Composition API
و TypeScript‌اند؛ برای استکی که `any` را ممنوع کرده، این یعنی لبه‌های تیز.
دو سیستم theming یعنی agent باید دو مدل ذهنی یاد بگیرد. و RTL آن خوب است
اما اکوسیستم فارسی/جلالی‌اش نازک‌تر است.

**پل:** و کاندیدای چهارم، که اصلاً کتابخانه نیست.

## اسلاید ۱۱ — کاندیدای ۴: shadcn-vue

**روی صفحه:** کارت قوت‌ها / هشدارها.

**متن ارائه:**
shadcn-vue پارادایم را عوض می‌کند: **نصب نمی‌کنید.** CLI سورس هر کامپوننت
را داخل repo شما کپی می‌کند. روی همان primitiveهای Reka UI ساخته شده که
Nuxt UI هم استفاده می‌کند، به‌علاوه‌ی Tailwind.

برای توسعه‌ی agent-driven این تقریباً رویاست: کل کد کامپوننت جلوی چشم
agent است؛ می‌تواند بخواند، بفهمد و patch کند. هیچ lock-in و هیچ آپگرید
اجباری‌ای هم وجود ندارد. theming هم ذاتاً CSS variable است.

اما همان مزیت، هزینه است: **مالکیت یعنی نگهداری.** هر باگ‌فیکس upstream
را باید خودمان بیاوریم. جدول داده ندارد — باید با TanStack Table سرهمش
کنید. کامپوزیت‌های آماده کم است و پروژه پورت جامعه است، نه محصول رسمی.

برای تیمی که می‌خواهد ۸۰٪ نیازش را از قفسه بردارد و روی دامنه تمرکز کند،
این trade-off نمی‌صرفد. ولی فلسفه‌اش را نگه می‌داریم — چند اسلاید دیگر
می‌بینید کجا.

**پل:** همه‌ی کاندیداها را دیدیم. برویم سراغ جدول امتیاز.

## اسلاید ۱۲ — جدول امتیاز

**روی صفحه:** ماتریس شش معیار × چهار فریم‌ورک با نقطه‌های امتیاز؛ سطر جمع؛ اشاره به Quasar و Element Plus.

**متن ارائه:**
این جدول جمع‌بندی چهار اسلاید قبل است. چند الگو در آن روشن است:

- در **عمق کامپوننت**، Vuetify و PrimeVue پنج می‌گیرند — این واقعاً
  زمین بازی آن‌هاست.
- در **مالکیت و theming توکنی**، shadcn-vue و Nuxt UI جلوترند.
- اما در دو معیاری که برای use case ما وزن بیشتری دارند — **توکن‌محوری** و
  **agent-friendliness** — به‌علاوه‌ی پوشش کامل RTL و کامپوننت کافی،
  Nuxt UI تنها کاندیدایی است که هیچ‌جا زیر چهار نمی‌گیرد. جمع: ۲۸ در برابر
  ۲۵ و ۲۵ و ۲۴.

دو اسم غایب را هم جواب بدهم: **Quasar** فریم‌ورک همه‌کاره‌ی خوبی است اما
دنیای CLI و build خودش را دارد — ما تازه از یک فریم‌ورک چتری بیرون آمده‌ایم.
**Element Plus** هم کلاسیک پنل‌های ادمین است اما داستان RTL آن ضعیف‌تر از
حد نیاز ماست. به همین دلیل به مقایسه‌ی نهایی نرسیدند.

**پل:** جدول جهت را نشان می‌دهد؛ حکم را صریح اعلام کنیم.

## اسلاید ۱۳ — حکم: Nuxt UI v4 standalone

**روی صفحه:** بنر حکم + پنج دلیل شماره‌دار.

**متن ارائه:**
انتخاب ما: **Nuxt UI نسخه‌ی ۴، در حالت standalone، روی Vue خالص.**

پنج دلیل، به ترتیب اهمیت:

۱. **این‌جا اثبات شده.** front-cp-v2 با همین ترکیب در production است؛
ریسک ناشناخته نداریم.

۲. **خط لوله‌ی توکن ما بومی آن است.** tokens.css با Tailwind @theme دقیقاً
مدل theming خود فریم‌ورک است — نه لایه‌ی ترجمه لازم است نه fork.

۳. **برای agentها آماده است.** مستندات llms.txt، skill رسمی pin شده در
skills-lock.json، و API یکدست TypeScript.

۴. **یک کتابخانه برای همه‌چیز.** از فرم تا جدول تا toast — بدون mix کردن
چند کتابخانه و چند زبان طراحی.

۵. **فلسفه‌ی shadcn را هم نگه داشته‌ایم.** همه‌ی مصرف‌ها از پشت
wrapperهای `Base*` خودمان می‌گذرد؛ سطح API مال ماست و اگر روزی لازم شد،
زیرش قابل تعویض است.

این تصمیم امروز بسته می‌شود، مگر اعتراض جدی‌ای باشد.

**پل:** فریم‌ورک انتخاب شد. حالا نظم — اول نظم کامپوننت‌ها.

**متن ارائه:**
پرده‌ی سوم. هدفش یک جمله است: هیچ‌کس — نه انسان، نه agent — هرگز نپرسد
«این کامپوننت را کجا بسازم؟». چهار لایه و یک درخت تصمیم، و ابهام صفر.

**پل:** اول چهار لایه.

---

## اسلاید ۱۵ — چهار لایه

**روی صفحه:** پشته‌ی چهار لایه: `Base*`، `App*`، کامپوننت‌های ماژول، screenهای ماژول.

**متن ارائه:**
از پایین به بالا:

- **`Base*`** در `shared/ui/base/` — primitiveهای سراسری. این‌ها Nuxt UI و
  کتابخانه‌های سنگین را wrap می‌کنند. فقط props و emits؛ نه store، نه API.
  مثل `BaseInput`، `BaseDataTable`، `BaseFilePond`.
- **`App*`** در `shared/ui/shell/` — پوسته‌ی اپلیکیشن: هدر، سایدبار،
  منوی کاربر. این لایه layout را می‌شناسد ولی دامنه را نه.
- **کامپوننت‌های ماژول** — UI گنگِ مخصوص دامنه. props دامنه‌شکل می‌گیرند
  (مثلاً `MediaCard` یک Media می‌گیرد) اما هنوز گنگ‌اند: نه store، نه fetch.
- و **screenها** — تنها لایه‌ی هوشمند. use-case و store را به هم وصل
  می‌کنند و داده را به پایین پاس می‌دهند.

قانون طلایی پایین اسلاید: **فقط screenها هوشمندند. بقیه فقط props را
render می‌کنند.** این قانون همان چیزی است که تست‌پذیری و قابل‌استفاده‌بودن
مجدد را تضمین می‌کند.

**پل:** حالا الگوریتم تصمیم — به‌شکل درخت.

---

## اسلاید ۱۶ — کامپوننت جدید کجا می‌رود؟

**روی صفحه:** فلوچارت Mermaid با چهار سؤال؛ پانوشت قرارداد `data-test` و سقف خطوط.

**متن ارائه:**
این درخت، عیناً در ruleهای agent هم کدنویسی شده. چهار سؤال به ترتیب:

۱. کتابخانه‌ی شخص ثالث را wrap می‌کند؟ → `Base*`.
۲. پوسته‌ی اپ است (ناوبری، هدر)؟ → `App*`.
۳. دو ماژول یا بیشتر استفاده‌اش می‌کنند و منطق دامنه ندارد؟ → باز هم `Base*`.
۴. به store یا use-case نیاز دارد؟ → Screen. وگرنه → کامپوننت گنگ ماژول.

و دو قرارداد ثابت که lint و review هر دو چک می‌کنند: هر المان تعاملی یک
`data-test` پایدار دارد (تست‌های browser به همین می‌چسبند)، و سقف ۸۰ خط
template و ۶۰ خط script — از آن به بعد باید یا composable جدا کرد یا
sub-component.

**پل:** روی مورد اول درخت — wrap کردن کتابخانه‌های سنگین — یک اسلاید
مکث می‌کنیم، چون پرتکرارترین اشتباه همان‌جاست.

## اسلاید ۱۷ — کتابخانه‌های سنگین پشت wrapper

**روی صفحه:** سه کارت: BaseFilePond، BaseRichTextEditor، BaseServerDataTable + سه دلیل.

**متن ارائه:**
FilePond برای آپلود، TipTap برای ویرایشگر متن، و جدول‌های سنگین — هیچ‌کدام
هرگز مستقیم در feature استفاده نمی‌شوند. همیشه پشت یک wrapper پروژه‌ای:

- `BaseFilePond` آپلود resumable با tus را یک‌جا پیکربندی می‌کند.
- `BaseRichTextEditor` تصمیم toolbar و extensionها را یک‌بار می‌گیرد.
- `BaseServerDataTable` جدول را با state صفحه‌بندی و sort سمت سرور و
  همگام‌سازی با URL جمع می‌کند.

چرا؟ سه دلیل: مصرف‌کننده هرگز API فروشنده را import نمی‌کند، پس تعویض یا
آپگرید کتابخانه فقط یک فایل را عوض می‌کند. و برای agentها: به‌جای پنج
داکیومنت متفاوت فروشنده، فقط **یک API پروژه‌ای** را یاد می‌گیرند.

این همان جایی است که گفتم فلسفه‌ی shadcn را نگه می‌داریم: کدِ سطح تماس،
مال ماست.

**پل:** کامپوننت‌ها منظم شدند. حالا سؤال سخت‌تر: منطق کجا زندگی کند؟

## اسلاید ۱۸ — پرده‌ی چهارم: منطق کجا زندگی می‌کند

**روی صفحه:** جداکننده‌ی پرده‌ی IV.

**متن ارائه:**
پرده‌ی چهارم. سه کلمه‌ای که بیشترین بحث‌های review ما را می‌سازند:
composable، store، util. هر سه «جایی برای گذاشتن منطق»اند و دقیقاً به همین
دلیل مرزشان باید شفاف باشد — وگرنه همه‌چیز یا composable می‌شود یا store.

**پل:** اول تعریف دقیق هر کدام.

## اسلاید ۱۹ — سه خانه، هر کدام یک قانون

**روی صفحه:** سه کارت تعریف + کارت use-case ماژول.

**متن ارائه:**
- **Composable** یعنی *رفتار reactive قابل استفاده‌ی مجدد*. تابعی که ref و
  computed و متد برمی‌گرداند. نکته‌ی حیاتی: state آن مال صدازننده است — هر
  بار صدا بزنی یک نسخه‌ی تازه می‌گیری. composable حافظه‌ی سراسری **ندارد**.
- **Store (Pinia)** یعنی *state مشترکی که از route عبور می‌کند*. singleton
  است: همه‌جای اپ همان یک نسخه را می‌بینند. devtools و persistence دارد.
  و مهم‌تر از همه: store باید **استثنا** باشد، نه پیش‌فرض.
- **Util** یعنی *تابع خالص*. نه reactivity، نه import از Vue. ورودی یکسان،
  خروجی یکسان — و به همین دلیل ساده‌ترین چیز برای unit test.

و شکل چهارم که در واقع همان composable است اما با آدرس مشخص:
**use-case ماژول** در `application/use-cases/` — ارکستراسیون یک جریان
دامنه: fetch به‌علاوه‌ی قواعد به‌علاوه‌ی state برای screen خودش.

**پل:** تعریف کافی است؛ الگوریتم انتخاب را ببینیم.

---

## اسلاید ۲۰ — کدام‌یک؟ به ترتیب بپرس

**روی صفحه:** فلوچارت Mermaid سه‌سؤاله + پانوشت «فقط ۳ store سراسری».

**متن ارائه:**
ترتیب سؤال‌ها عمدی است — از محدودترین شروع می‌کنیم:

۱. **تابع خالص است و reactivity نمی‌خواهد؟** util در `shared/lib/`. تمام.
۲. **state باید از route عبور کند یا بین ماژول‌ها مشترک باشد؟** store.
و فقط همین حالت store می‌شود.
۳. **جریانِ یک دامنه‌ی مشخص است؟** use-case همان ماژول.
۴. وگرنه: **composable مشترک** در `shared/composables/`.

و آن عدد پایین اسلاید مهم‌ترین عدد امروز است: کل `front-cp-v2` فقط
**سه store سراسری** دارد — auth، theme و cart. نه ده‌تا، نه سی‌تا. اگر
store هایتان زیاد می‌شوند، معماری دارد فریاد می‌زند که state محلی را
سراسری کرده‌اید. بقیه‌ی state یا مال use-case است یا مال URL.

**پل:** بیایید همین قواعد را روی کد واقعی ببینیم.

## اسلاید ۲۱ — مثال‌های واقعی از front-cp-v2

**روی صفحه:** جدول artifact / نوع / دلیل.

**متن ارائه:**
- `useCustomFetch` — composable مشترک. تنها دروازه‌ی HTTP اپ. refresh توکن
  و retry بعد از 401 با محافظ single-flight — یعنی اگر ده request همزمان
  401 بخورند، فقط یک refresh انجام می‌شود. این‌ها یک‌بار نوشته شده و همه
  استفاده می‌کنند.
- `useValidationRules` — یک واژگان اعتبارسنجی برای کل اپ، متصل به i18n.
  هیچ regex ای inline نوشته نمی‌شود.
- `useSnackbar` — یک facade روی toast کتابخانه‌ی UI. اگر روزی Nuxt UI را
  عوض کنیم، فقط همین فایل عوض می‌شود — مصرف‌کننده‌ها دست نمی‌خورند.
- سه store سراسری: `auth` (نشست)، `theme` (حالت تاریک)، `cart` (سبدی که
  بین ماژول‌ها مشترک است). هر سه واقعاً state مشترک‌اند.
- `shared/lib/date.ts` — util خالص تبدیل جلالی/میلادی؛ بدون Vue و با
  unit test کامل.
- `useCampaignListScreen` — نمونه‌ی use-case: کل جریان یک صفحه‌ی لیست —
  فیلترها، fetch، state جدول — در یک فایل کنار screen.

**پل:** و حالا روی دیگر سکه — الگوهایی که مستقیم رد می‌شوند.

## اسلاید ۲۲ — ردشدنی‌های فوری

**روی صفحه:** چهار کارت قرمز.

**متن ارائه:**
چهار الگو که در review بدون بحث رد می‌شوند — و ESLint هم بیشترشان را
همان لحظه‌ی ذخیره می‌گیرد:

۱. **fetch داخل کامپوننت گنگ.** کامپوننت props را render می‌کند؛ داده
آوردن کار use-case است.
۲. **store به‌عنوان کش همه‌چیز.** state ای که فقط مال یک صفحه است در
use-case یا URL زندگی می‌کند، نه در Pinia.
۳. **util ای که Vue import می‌کند.** همان لحظه‌ای که به ref نیاز پیدا کرد،
دیگر util نیست — composable است؛ جابه‌جایش کنید.
۴. **HTTP client دوم.** همه‌ی ترافیک از `useCustomFetch` می‌گذرد. axios یا
fetch خام یعنی auth و خطا و retry از policy مرکزی خارج شده.

**پل:** کد منظم شد. حالا زیبایی — پرده‌ی آخر: سیستم طراحی.

پرده‌ی پنجم. سیستم طراحی ما نه PDF است و نه صفحه‌ی Figma ای که خاک

## اسلاید ۲۴ — design tokenها

**روی صفحه:** ردیف‌های swatch رنگی، قطعه‌ی tokens.css، نگاشت اسم‌های Figma.

**متن ارائه:**
design token یعنی: هر رنگ، فونت و فاصله فقط در **یک فایل** تعریف می‌شود —
`assets/css/tokens.css` — و بقیه‌ی اپ فقط با *اسم معنایی* به آن ارجاع
می‌دهد.

زنجیره سه حلقه دارد: **پالت خام** (brand-50 تا brand-950 و رنگ‌های state)،
**توکن معنایی** (مثل «سطح ۲ سطح خنثی»)، و **کلاس utility** که در کامپوننت
مصرف می‌شود.

دو نتیجه‌ی عملی:

- **نگاشت Figma یک‌به‌یک است.** طراح می‌گوید `Content/Neutral/7`؛ agent
  بدون حدس‌زدن می‌نویسد `text-content-neutral-level-7`. این نگاشت خودش یک
  skill مکتوب است.
- **حالت تاریک رایگان است.** زیر کلاس `.dark` فقط *مقدار* توکن‌ها عوض
  می‌شود؛ حتی یک کامپوننت هم دست نمی‌خورد.

و یادتان هست؟ این ساختار دقیقاً مدل بومی theming در Nuxt UI است — دلیل
شماره‌ی ۲ حکم پرده‌ی دوم.

**پل:** توکن‌ها منبع حقیقت‌اند؛ اما کجا می‌شود *دیدشان*؟ Labs.

## اسلاید ۲۵ — Labs، کاتالوگ زنده

**روی صفحه:** دو کارت (چیستی / نحوه‌ی استفاده) + چرخه‌ی توسعه به‌شکل فلوچارت.

**متن ارائه:**
Labs یک route واقعی در خود اپ است — `/labs` — و یک ماژول واقعی. هر
خانواده‌ی primitive یک سکشن دارد: دکمه‌ها، input ها، جدول‌ها، date picker
جلالی، آپلود، chip ها… هر `Base*` با همه‌ی variant هایش، در هر دو تم.

دو قانون درباره‌اش داریم:

۱. **agent قبل از ساختن هر UI جدید، اول Labs و `shared/ui/base` را
می‌بیند.** این قانون در rule ها مکتوب است و جلوی دوباره‌سازی primitive ها
را می‌گیرد.
۲. **هر Base جدید، در همان PR سکشن Labs خودش را می‌گیرد.** کامپوننت بدون
ویترین merge نمی‌شود.

چرخه‌ی توسعه‌ی Labs هم روی اسلاید است: کامپوننت را بساز یا تغییر بده →
سکشن Labs را اضافه/به‌روز کن → با browser MCP در هر دو تم و حالت RTL چک
کن → بازخورد طراح را روی *صفحه‌ی در حال اجرا* بگیر، نه روی اسکرین‌شات →
ship. اگر بازخورد اصلاحی بود، برمی‌گردیم به قدم اول.

برای طراح‌ها هم Labs همان زبان مشترک است: چیزی که در Labs است، حقیقتِ
محصول است.

**پل:** و حالا نخ تسبیح کل ارائه.

## اسلاید ۲۶ — هیچ‌کدام به حافظه تکیه ندارد

**روی صفحه:** فلوچارت: rules → skills/generated docs → agent → hooks → ship.

**متن ارائه:**
اگر از کل امروز یک چیز در خاطرتان بماند، این اسلاید باشد.

هر تصمیمی که امروز گرفتیم — فریم‌ورک UI، چهار لایه‌ی کامپوننت، درخت
تصمیم منطق، قانون Labs — **در قالب rule و skill کدنویسی می‌شود**، نه صفحه‌ی
ویکی و نه حافظه‌ی جمعی.

جریان این است: قواعد canonical در `.claude/rules/` نگه‌داری می‌شوند؛
`CLAUDE.md` و `AGENTS.md` از روی آن‌ها generate می‌شوند و hook محافظ اجازه‌ی
ویرایش مستقیم‌شان را نمی‌دهد — پس Claude و Codex همیشه یک روایت واحد
می‌بینند. برای هر نوع artifact — کامپوننت، composable، store، screen —
یک skill نویسندگی هست که الگوی درست را قدم‌به‌قدم می‌دهد. و در انتهای هر
کار، hookها gate می‌کنند: lint، typecheck، تست، وجود `data-test` و کامل
بودن هر دو زبان. اگر رد شود، agent باید درست کند — بدون استثنا.

نتیجه: نفر بعدی — انسان یا agent — دقیقاً همین الگوها را اجرا می‌کند و
معماری drift نمی‌کند.

**پل:** جمع‌بندی و قدم‌های بعد.

## اسلاید ۲۷ — پنج تصمیم

**روی صفحه:** پنج تصمیم شماره‌دار + قدم‌های بعدی.

**متن ارائه:**
جمع‌بندی در پنج خط:

۱. **پایه: Vue خالص + Vite.** صراحت به‌جای جادو؛ هر جایگزین قابل خواندن.
۲. **فریم‌ورک UI: Nuxt UI v4 در حالت standalone.** توکن‌محور، RTL، آماده
برای agent — و اثبات‌شده در production خودمان.
۳. **کامپوننت‌ها در چهار لایه:** Base*، App*، کامپوننت ماژول، screen —
فقط screen هوشمند است.
۴. **منطق با درخت تصمیم:** util خالص → store فقط برای state واقعاً مشترک →
use-case برای جریان دامنه → composable مشترک.
۵. **سیستم طراحی زنده:** tokens.css منبع حقیقت، Labs ویترین اجباری.

قدم‌های بعدی به ترتیب: اسکافولد repo با همین استک؛ پورت `tokens.css`؛
**اول Labs را می‌سازیم** — چون همه‌ی primitive ها را همان‌جا تثبیت می‌کند —
و بعد اولین ماژول محصول.

ممنون. سؤال‌ها؟
